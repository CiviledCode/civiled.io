Shellcoding

August 31, 2024 - 2 mins read
  • shellcode
  • technique
  • execute

Shellcoding is the technique of placing machine code within an RWX (Read-Write-Execute) segment of memory and then jumping execution to it. Shellcoding is possible under two primary circumstances:

  1. RWX Segments with a Write Primitive: There are RWX segments in memory, and you have a primitive that allows you to write into these segments.

  2. Writing to Virtual Memory via /proc/self/mem: There is a primitive that allows you to write directly to virtual memory through interfaces like /proc/self/mem. This file represents the memory of the process and can be used to modify the process’s memory space, potentially allowing the injection of shellcode.

In both cases, once the shellcode is successfully placed in memory, the next step is to redirect the program’s execution flow to the shellcode, typically by modifying a return address or exploiting another control flow vulnerability. This list may not be exhaustive.

Protections

Modern binaries and operating systems employ several protections to prevent shellcoding, making it more challenging for attackers to inject and execute arbitrary code:

  1. Data Execution Prevention (DEP) / NX Bit: DEP, often implemented using the NX (No-Execute) bit, prevents code execution in memory regions marked as non-executable. This means that even if an attacker can inject shellcode into a writable region of memory, they won’t be able to execute it.

Bypass: If other code execution techniques are possible, a carefully constructed payload can be used to call the mprotect syscall:

/*
    mprotect() changes the access protections for the calling
    process's memory pages containing any part of the address range
    in the interval [addr, addr+len-1].  addr must be aligned to a
    page boundary.

    If the calling process tries to access memory in a manner that
    violates the protections, then the kernel generates a SIGSEGV
    signal for the process.
*/
int mprotect(void *addr, size_t len, int prot);

In order to change the protections on the segment, you would need a leak of the address of that segment. It’s imperative that the addr is page-aligned.

In some kernels, there might be additional protections in place that limit what permissions mprotect is allowed to set for a memory region.

Additionally, access control systems like Seccomp and AppArmor configurations might impede the ability to effectively use mprotect.
